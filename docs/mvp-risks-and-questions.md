# MVP risks and open questions

## Главные риски

1. Нестабильные AI-ответы
   - Модель может возвращать слишком общий текст или смешивать роли.
   - Снижение риска: строгие prompt templates, Zod validation, повторные retries, сохранение raw response.

2. Смена моделей и API
   - OpenRouter и модели генерации могут менять возможности.
   - Снижение риска: model registry и provider adapter, чтобы UI-узлы не зависели от конкретной модели.

3. Большие изображения
   - Base64 быстро увеличивает payload, generation/analyze calls могут стать тяжелыми.
   - Снижение риска: сжимать preview, ограничивать размер upload, downscale перед отправкой в AI.

4. Сложность графа
   - Если сразу добавить много типов узлов, продукт станет трудно отлаживать.
   - Снижение риска: начать с 5-6 node types и одной вертикали.

5. Хранение файлов в браузере
   - IndexedDB подходит для прототипа, но не для командной production-системы.
   - Снижение риска: сразу отделить metadata от asset storage, чтобы позже заменить IndexedDB на S3/MinIO.

6. Конфликтующие референсы
   - Пользователь может попросить "стиль отсюда", "героя отсюда", "не брать одежду", и модель смешает лишнее.
   - Снижение риска: explicit roles, weights, negative constraints, preview финального prompt перед генерацией.

7. Производительность канваса
   - Много изображений и связей могут тормозить UI.
   - Снижение риска: thumbnails, lazy preview, memoized node cards, не хранить base64 прямо в graph JSON.

## Вопросы перед активной разработкой

1. Какой самый первый demo-сценарий считаем успехом?
   - Например: "загрузить 2 референса, взять стиль из первого, композицию из второго, сгенерировать новую обложку".

2. Какие 3-5 node types нужны в самом первом экране?
   - Мой вариант: Import Image, Image to Text, Text Prompt, Reference Composer, Generate Image, Preview.

3. На каком языке должны быть AI-results и prompts?
   - UI может быть русским, но для части image models английский prompt часто стабильнее.

4. Какую модель генерации хотим использовать первой?
   - В макете написано `NanoBanana`, но нужно подтвердить конкретное имя модели в OpenRouter и формат image input/output.

5. Нужно ли сохранять все intermediate results?
   - Например, каждый анализ, каждый prompt composer output и каждую генерацию как run history.

6. Нужен ли project export/import уже в MVP?
   - Это полезно, если прототип будет переноситься между машинами без backend.

7. Насколько строго нужно копировать визуальный стиль Figma?
   - Можно стартовать 1:1 с минимальными белыми карточками, а панели/инспектор взять из prodSQL-паттернов.

8. Нужно ли сразу делать inspector справа?
   - Альтернатива: настройки внутри карточек, как в текущем Figma-макете. Inspector удобнее, когда настроек станет много.

9. Какие роли референса являются обязательными для первой версии?
   - Я бы начал с `style`, `composition`, `actors/actions`, `lighting`, `color`, `ocr`.

10. Есть ли лимит бюджета на OpenRouter experiments?
    - Нужен хотя бы ручной предохранитель: "не запускать batch/variations без подтверждения".
